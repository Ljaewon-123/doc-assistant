## 1. CLAUDE.md 작성 원칙

### ✅ Lazy Loading 구조 사용

CLAUDE.md에 모든 내용을 직접 쓰지 않는다. **참조만 두고 상세 내용은 별도 파일로 분리**한다.
매 세션마다 불필요한 토큰 소비를 방지하고, Claude가 필요할 때만 해당 파일을 읽게 한다.

```markdown
# CLAUDE.md

## 프로젝트 문서
- API 스펙: @docs/api-spec.md
- DB 스키마: @docs/db-schema.md
- 코딩 컨벤션: @docs/conventions.md
- 아키텍처: @docs/architecture.md
```

### ✅ 폴더별 CLAUDE.md 분리

기능 단위로 CLAUDE.md를 분리하면 해당 폴더 작업 시에만 로드된다.

```
src/auth/CLAUDE.md      ← 인증 관련 규칙만
src/payments/CLAUDE.md  ← 결제 관련 규칙만
```

### ✅ Mermaid로 아키텍처 정리

Claude가 프로젝트 구조를 한눈에 파악할 수 있도록 다이어그램을 포함한다.
상세 내용은 `docs/architecture.md`에 작성하고 CLAUDE.md에서 참조.

```
graph LR
    A[Client] --> B[API Gateway]
    B --> C[Auth Service]
    B --> D[Core Service]
    D --> E[DB]
```

## 2. 컨텍스트 관리

### 핵심 원칙: 한 세션 = 한 피처

하나의 세션에서 여러 기능을 연달아 구현하지 않는다.
신선한 컨텍스트가 부풀어진 컨텍스트보다 항상 낫다.

| 명령어 | 용도 |
|--------|------|
| `/clear` | 컨텍스트 완전 초기화 |
| `/compact` | 대화 압축 (맥락 유지) |
| `/context` | 현재 토큰 사용량 확인 |
| `/memory` | 자동 학습된 메모리 확인·편집 |
| `/mcp` | MCP 서버 활성화·비활성화 |

### 세션 워크플로우

```
Plan Mode에서 전체 작업 설계 및 단계 분리
    ↓
/clear 또는 새 세션 열기
    ↓
첫 번째 단계만 구현
    ↓
테스트 → 린트 → 커밋
    ↓
/clear → 다음 단계
```

### MCP 토큰 관리

- `/context`로 주기적으로 토큰 사용량 확인
- 현재 작업에 불필요한 MCP는 `/mcp`에서 비활성화
- Notion, Linear 등 도구 설명이 큰 MCP는 필요할 때만 활성화

### 무거운 작업은 스크립트로 오프로드

무거운 데이터를 대화 안에서 직접 처리하면 컨텍스트가 오염된다.
**Claude에게 스크립트를 작성하게 하고, 결과 요약만 받는다.**

| 상황 | 방법 |
|------|------|
| 대용량 CSV 검증 | 파싱 스크립트 작성 → 결과를 `summary.json`으로 |
| 로그 분석 | 에러 패턴 추출 스크립트 → `report.md`로 |
| API 응답 비교 | diff 스크립트 작성 → `diff.json`으로 |

## 3. 코딩 워크플로우

### Plan Mode 먼저 (`Shift+Tab`)

큰 변경 작업은 반드시 Plan Mode로 시작한다.

```
1. Plan Mode에서 작업 설명
2. Claude가 수정할 파일·접근 방식 계획 제시
3. 계획 리뷰 및 피드백
4. 승인 후 Accept Mode로 전환하여 실행
```

> Plan 없이 바로 실행하면 Claude가 엉뚱한 방향으로 코드를 대량 수정하는 사태가 발생한다.

### TDD 기반 반복 루프

```
기능 구현 → 테스트 → 린트/포맷 → 커밋 → 반복
```

문제가 생겨도 마지막 커밋으로 롤백하면 되므로 디버깅이 쉬워진다.

### Thinking 로그 읽기

Claude의 추론 과정을 무시하지 않는다.
**잘못된 가정을 발견하는 즉시 `Escape`로 중단**한다.
잘못된 가정 위에 쌓인 코드는 전부 쓸모없다.

### 에러 처리

에러를 해석해서 설명하지 않는다. **에러 로그를 그대로 붙여넣는다.**
직접 해석하면 오히려 정보가 누락될 수 있다.

### 막힐 때: 다른 AI에게 비평 받기

`/export`로 대화를 내보내고 ChatGPT나 Gemini에 전달한다.

> "이 대화를 분석해서 Claude가 놓치고 있는 것이나 잘못된 접근이 있으면 지적해줘"

## 4. 작업 연속성 관리

### TODO.md로 태스크 추적

프로젝트 루트에 `TODO.md`를 만들고 Claude가 상태를 업데이트하게 한다.

```markdown
## 진행 중
- [ ] 결제 기능 구현 (Stripe 연동)
- [ ] 구독 시스템 설계

## 완료
- [x] 인증 모듈 구현
- [x] DB 스키마 마이그레이션
```

**실전 사용법:**
1. 하루 시작 시 할 일을 TODO.md에 체크리스트로 작성
2. Claude에게 `"TODO.md 읽고 첫 번째 항목부터 시작해"` 지시
3. 세션 종료 시 `"TODO.md 업데이트해줘"` → 진행 상황 자동 반영

### /memory 활용 (Second Brain)

프로젝트에서 발견한 중요한 결정 사항을 기록한다.
`"기억해줘"`라고 말하면 저장되고 `/memory`로 확인·편집 가능.

```markdown
## 2025-01-15: 인증 방식 변경
- JWT → 세션 기반으로 변경
- 이유: 모바일 앱 토큰 리프레시 이슈
- 참고: src/auth/session.ts
```

> 저장 위치: `~/.claude/projects/<project>/memory/MEMORY.md`
> 매 세션 시작 시 처음 200줄 자동 로드

## 5. WAT 프레임워크 (복잡한 프로젝트용)

복잡한 기능을 구현할 때 적용하는 3단계 방법론.

### W — Workflow 먼저 정의

코드를 쓰기 전에 작업 흐름을 글로 정의한다. 10분 투자로 수시간의 삽질을 줄인다.

```
"댓글 기능을 추가해줘. 다음 순서로 진행해:
1. comments 테이블 스키마 설계 및 마이그레이션
2. 댓글 CRUD API 구현
3. 프론트엔드 댓글 컴포넌트 구현
4. 각 단계마다 테스트 작성 및 통과 확인"
```

### A — Agent 병렬 처리

Sub-agent로 역할을 분리해 병렬 처리한다.

```
Sub-agent A → 테스트 작성 및 실행
Sub-agent B → 관련 문서 업데이트
Sub-agent C → 코드 린트 및 타입 체크
```

에러 발생 시 Self-healing으로 스스로 로그를 읽고 수정 후 재실행.

### T — Tools 원자적 분리

하나의 거대한 스크립트보다 **작고 원자적인 도구들의 조합**이 낫다.

```bash
# ❌ deploy-all.sh (200줄짜리)
# ✅ 원자적 도구
scripts/build.sh      # 빌드만
scripts/test.sh       # 테스트만
scripts/migrate.sh    # 마이그레이션만
scripts/deploy.sh     # 배포만
```
